第一个问题

Q: 让我们来讨论一个问题: 在AI时代如何学习CS? 或许, 第一个应该被提出的问题是还有必要学习吗? 最近我在各种社交媒体上不断的看到所谓的Vibe Coding, 有些人完全不懂html, css等, 但做出了很好的网站, 也有完全不懂计算机编程语言的人做出了小游戏, 或者具有交互效果的动画, 遵循传统路径, 要完成这些事情, 其实需要一定的时间(当然, 我这里说的不管网页还是小游戏等都是具备一定完成度的, 如果单纯只是想做一个网页, 其实写一个html文件就好了, 那花不了多长时间.). 可是现在, 我们已经有诸如Codex, Claude Code等工具, 有些时候真的只是几个指令的问题. 基于这种情况, 你觉得学习计算机科学, 或者我们先具体一点, 学习计算机编程语言还有意义吗?

Q: Let’s discuss a question: How should we learn Computer Science (CS) in the era of AI? Perhaps the first question that ought to be raised is whether it is even necessary to learn it anymore. Recently, I’ve been constantly seeing the term “Vibe Coding” across various social media platforms. Some people who know absolutely nothing about HTML, CSS, or other technologies have built excellent websites. Similarly, individuals with zero knowledge of computer programming languages have created mini-games or interactive animations. Following a traditional path, achieving these things would actually take a considerable amount of time. (Of course, the webpages or mini-games I’m referring to here all have a certain degree of completeness. If one simply wanted to make a basic webpage, writing a single HTML file would suffice, which wouldn’t take much time at all.) However, nowadays, we already have tools like Codex and Claude Code; sometimes it really is just a matter of a few prompts. Given this situation, do you think studying computer science—or to be more specific for now, learning computer programming languages—still makes sense?

A: 是的——学习计算机科学（CS）依然有意义，但原因已经变了。

在AI出现之前，学习编程通常是必不可少的，因为你必须亲手敲出那些“咒语”。HTML、CSS、JavaScript、Python、Git、终端命令——没有这些，什么都运转不起来。

现在，AI可以替你敲出大部分咒语。像 Codex 这样的工具被明确设计为编码智能体（coding agents），它们可以在真实的仓库中运行，无缝嵌入本地、编辑器或云端的工作流，并在人类的监督下承担更大的任务。OpenAI 将这种新型的 Codex 工作流描述为开发者“在各个项目中编排多个智能体”，而不仅仅是亲手编写每一行代码。同样，Claude Code 以及 Claude 的开发者工具也指向了这种智能体化的编程工作流，模型可以使用各种工具，并通过 API 和代码环境来协助构建应用程序。

因此，过去的问题是：

“我会写这段代码吗？”

现在的新问题则是：

“我能理解、指挥、评判、修复并掌控这段代码吗？”

这第二个问题依然极具现实意义。

Yes — learning CS still makes sense, but the reason has changed.

Before AI, learning programming was often necessary because you had to personally type the spell. HTML, CSS, JavaScript, Python, Git, terminal commands — without those, nothing moved.

Now AI can type many spells for you. Tools like Codex are explicitly designed as coding agents that can operate on real repositories, run in local/editor/cloud workflows, and take on larger tasks with human oversight. OpenAI describes newer Codex workflows as developers “orchestrating multiple agents across projects,” not merely writing every line by hand. Claude Code / Claude’s developer tools similarly point toward agentic programming workflows, where the model can use tools and help build applications through APIs and code environments.

So the old question was:

“Can I write the code?”

The new question is:

“Can I understand, direct, judge, repair, and own the code?”

That second question is still very much alive.

Vibe Coding证明了编程变得更加触手可及——而不是变得毫无必要

我认为“vibe coding”是真实存在的。一个几乎没有技术背景的人可以说：

  • 帮我为我的家教服务做一个落地页。
  • 加一个价格区域。
  • 加点动画。
  • 让它适配移动端。
  • 部署上线。

然后，轰的一声——他们可能就会得到一个完整度惊人的东西。

这并不是假的。它确实非常强大。vibe coding这个概念最初是由Andrej Karpathy在2025年初推广开来的，指的是一种你只需描述自己想要什么、然后让 AI 生成代码的开发风格。

但是在以下两者之间，存在着一个隐秘的鸿沟： > “我做了一个网站”

以及

“我把这个网站当作一个系统来理解。”

第一种现在很廉价，而第二种依然非常有价值。

一个什么都不懂的人也能建一个网站。但当某些东西崩溃时，他们可能不知道：

  • 为什么布局在手机上塌陷了？
  • 为什么这个按钮没反应？
  • 为什么部署失败了？
  • 为什么页面这么慢？
  • 为什么用户数据泄露了？
  • 为什么 AI 偷偷加了五个没必要的依赖项？
  • 为什么改了一个地方，另一个页面却莫名其妙地坏了？

这时候，计算机科学/编程知识就会重新推门走进来，对你说：“嗨，你在叫我吗？”

“Vibe Coding” proves programming has become more accessible — not unnecessary

I think “vibe coding” is real. Someone with almost no technical background can say:

  • Make me a landing page for my tutoring service.
  • Add a pricing section.
  • Add animation.
  • Make it mobile friendly.
  • Deploy it.

And boom — they may get something surprisingly complete.

This is not fake. It is genuinely powerful. The original “vibe coding” idea was popularized by Andrej Karpathy in early 2025 as a style where you describe what you want and let the AI generate the code.

But there is a hidden difference between:

“I made a website”

and

“I understand this website as a system.”

The first one is now cheap. The second one is still valuable.

A person who knows nothing can build a website. But when something breaks, they may not know:

  • Why does the layout collapse on mobile?
  • Why does this button not work?
  • Why is the deployment failing?
  • Why is the page slow?
  • Why is user data exposed?
  • Why did the AI add five unnecessary dependencies?
  • Why did one change silently break another page?

That is where CS/programming knowledge re-enters the room like: “Hello, you rang?”

AI 降低了记忆语法的价值，但提升了判断力的价值

在过去，初学者在这些事情上浪费了大量时间：

  • “怎么把一个 div 居中？”
  • “读取文件的精确语法是什么？”
  • “在 JavaScript 里怎么写 for 循环？”
  • “怎么配置这个安装包？”

AI 帮我们消除了大部分这种痛苦。太棒了。老实说，慢走不送。过去很多初阶的编程教育，本质上就是伪装成“严谨”的职场霸凌和刁难。

但 AI 并没有消除对更高阶能力的需求：

  • 你仍然需要知道自己要的是什么。
  • 你仍然需要知道 AI 给的答案好不好。
  • 你仍然需要知道如何调试（debug）。
  • 你仍然需要知道什么叫“足够好”。
  • 你仍然需要理解各种权衡（tradeoffs）。

这就像数学里的计算器。 计算器让算术变得不再那么重要，但它并没有让数学变得毫无必要。事实上，一旦算术变得廉价，更高层次的数学思维反而变得更加重要。

AI 和编程的关系也是如此。 AI 让代码产出的成本更低。因此，需求定义、架构设计、调试、验证以及审美，反而变得更加重要。

AI reduces the value of syntax memory, but increases the value of judgment

In the past, beginners wasted enormous time on things like:

  • “How do I center a div?”
  • “What is the exact syntax for reading a file?”
  • “How do I write a for-loop in JavaScript?”
  • “How do I configure this package?”

AI crushes much of that pain. Wonderful. Good riddance, honestly. A lot of beginner programming education was basically hazing disguised as rigor.

But AI does not remove the need for deeper abilities:

  • You still need to know what you are asking for.
  • You still need to know whether the answer is good.
  • You still need to know how to debug.
  • You still need to know what “good enough” means.
  • You still need to understand tradeoffs.

This is like calculators in mathematics.

The calculator makes arithmetic less important. But it does not make mathematics unnecessary. In fact, once arithmetic becomes cheap, higher-level mathematical thinking becomes more important.

Same with AI and programming.

AI makes code production cheaper. Therefore, specification, architecture, debugging, verification, and taste become more important.

编程语言仍然值得学习，因为它们教会你精准

这是我最坚定的观点： > 编程语言不仅仅是告诉计算机要做什么的工具。它是一个训练器，用来让你的思维变得精确。

自然语言是模糊、柔软的。你可以说：

帮我把网站做得优雅一点。

但“优雅”是什么意思？

而程序会逼着你说： - 页眉需要固定。 - 最大宽度应该是 960 像素。 - 表单需要验证邮箱格式。 - 动画应该在滚动时触发。 - 数据库需要存储用户 ID、时间戳和支付状态。 - 如果没有记录，函数应当返回 None。

这不仅仅是“写代码”，这是被规训过的逻辑思维。

所以，是的，AI 可以写代码。但如果你学习编程，你学到的是如何将模糊的欲望转化为精确的结构。这种能力依然极其宝贵。 实际上，在 AI 时代，它可能会变得更加宝贵，因为一个能够表达精确结构的人，可以更好地指挥 AI。

糟糕的提示词：

给我做一个学习类 App。

更好的提示词：

构建一个包含三个视图的单页应用：课程、测试和复习。在本地存储用户进度。每个测试题目应包含四个选项、即时反馈和重试机制。保持状态模型简单：包含 currentLesson（当前课程）、score（得分）、mistakes（错题）和 completedLessons（已完成课程）。

第二个提示词就来自一个已经开始像程序员一样思考的人。

Programming languages are still worth learning because they teach precision

Here is my strongest opinion:

A programming language is not merely a tool for telling the computer what to do. It is a training device for making your own thoughts precise.

Natural language is soft. You can say:

Make the website elegant.

But what does elegant mean?

A program forces you to say:

  • The header should be fixed.
  • The max width should be 960px.
  • The form should validate email.
  • The animation should trigger on scroll.
  • The database should store user ID, timestamp, and payment status.
  • The function should return None if no record exists.

This is not just “coding.” This is disciplined thinking.

So yes, AI can write code. But if you learn programming, you learn how to convert vague desire into exact structure. That skill is still extremely valuable.

Actually, in the AI era, it may become more valuable, because the person who can express precise structure can command AI much better.

Bad prompt:

Make me a learning app.

Better prompt:

Build a single-page app with three views: lesson, quiz, and review. Store user progress locally. Each quiz question should have four choices, immediate feedback, and a retry mechanism. Keep the state model simple: currentLesson, score, mistakes, and completedLessons.

The second prompt comes from someone who has begun to think like a programmer.

真正的分水岭：玩具、产品与系统

对于小打小闹的东西，vibe coding足够了。

个人主页？可以。 一个小动画？可以。 一个简单的计算器？可以。 一个小游戏？通常可以。 一个教学演示 Demo？完全可以。

但一旦某个东西变成了一个“系统”，纯粹的vibe coding就会变得极其脆弱。

一个系统拥有内存/状态、用户、错误处理、安全性、更新迭代、版本控制、依赖项、性能问题以及未来的功能变更。

一个小游戏可以靠“氛围编程”捏出来。 但一个带有账户、支付、排行榜、多人联机、反作弊和数据存储的游戏平台呢？那是完全不同的巨兽。

一个静态的教学页面可以靠“氛围编程”做出来。 但一个带有学生进度、教师看板、支付、身份验证和数据分析的严肃在线学习平台呢？你需要工程级别的判断力。

界限就在这里：

AI 非常擅长制造“产物”（artifacts）。而计算机科学（CS）帮助你理解“系统”（systems）。

而这个世界是运行在系统之上的，而不仅仅是产物。

The real distinction: toy, product, system

For small things, vibe coding is enough.

  • A personal homepage? Yes.
  • A small animation? Yes.
  • A simple calculator? Yes.
  • A mini game? Often yes.
  • A teaching demo? Absolutely yes.

But once something becomes a system, pure vibe coding becomes fragile.

A system has memory, users, errors, security, updates, version control, dependencies, performance issues, and future changes.

A mini-game can be vibe-coded. A game platform with accounts, payments, rankings, multiplayer, anti-cheat, and data storage? Different beast.

A static teaching page can be vibe-coded. A serious online learning platform with student progress, teacher dashboards, payment, authentication, and analytics? You need engineering judgment.

This is the line:

AI is very good at producing artifacts. CS helps you understand systems.

And the world runs on systems, not just artifacts.

那么，你还应该学习编程语言吗？

应该——但不能再用老一套的方法了。

你不需要再这样学习编程：

  • 背诵每一个语法细节。
  • 痛苦地做几百道死板的人工练习题。
  • 从零开始纯手写一切。
  • 假装 Stack Overflow / AI / 官方文档不存在。

那种学习方式正在被淘汰。 相反，你应该这样学编程：

  • 从第一天起就使用 AI，但绝不要让 AI 变成一个黑盒子。
  • 去构建真实的东西。
  • 阅读 AI 生成的代码。
  • 弄清楚为什么每个部分会存在。
  • 动手去修改它。
  • 把它搞砸、弄坏。
  • 修复它。
  • 把它重构成更干净的代码。

你的目标不是成为一个人类编译器，你的目标是成为一个能够“用代码去思考”的人。

So should you learn programming languages?

Yes — but not in the old way.

You do not need to learn programming like this:

  • Memorize every syntax detail.
  • Suffer through hundreds of artificial exercises.
  • Write everything from scratch.
  • Pretend Stack Overflow / AI / documentation does not exist.

That way is becoming obsolete.

Instead, learn programming like this:

  • Use AI from day one, but do not let AI become a black box.
  • Build real things.
  • Read the generated code.
  • Ask why each part exists.
  • Modify it manually.
  • Break it.
  • Fix it.
  • Rebuild it cleaner.

The goal is not to become a human compiler. The goal is to become someone who can think with code.

但是，编程之外的“计算机科学”呢？

这才是事情变得有趣的地方。 学习编程语言只是入场券，计算机科学（CS）本身要庞大得多：

  • 算法
  • 数据结构
  • 计算机系统
  • 网络
  • 数据库
  • 操作系统
  • 安全性
  • 编译器
  • 分布式系统
  • AI / 机器学习
  • 数值计算
  • 人机交互

在 AI 时代，其中一些非但没有贬值，反而变得更加重要。

例如，假设 AI 帮你写了一个程序。你如何知道它运行起来高不高效？ 你需要算法。

假设 AI 构建了一个登录系统。你如何知道它安不安全？ 你需要安全性知识。

假设 AI 生成了一个数据库架构。你如何知道它以后会不会变成一场灾难？ 你需要数据库思维。

假设 AI 写了一个数值模拟。你如何知道结果在数学上是否稳定？ 你需要数值分析。

假设 AI 写了一个网页应用，在本地运行良好，但部署后就崩了。 你需要系统/网络/DevOps 的基础知识。

所以，计算机科学并不会消失。它的重心从“动手实现”转移到了“概念掌控”。

But what about “Computer Science” beyond programming?

This is where things get interesting.

Programming language learning is only the entrance. CS itself is bigger:

  • Algorithms
  • Data structures
  • Computer systems
  • Networks
  • Databases
  • Operating systems
  • Security
  • Compilers
  • Distributed systems
  • AI / machine learning
  • Numerical computation
  • Human-computer interaction

In the AI era, some of these become more important, not less.

For example, suppose AI writes a program for you. How do you know whether it is efficient? You need algorithms.

Suppose AI builds a login system. How do you know whether it is safe? You need security.

Suppose AI generates a database schema. How do you know whether it will become a disaster later? You need database thinking.

Suppose AI writes a numerical simulation. How do you know whether the result is mathematically stable? You need numerical analysis.

Suppose AI writes a web app that works locally but fails after deployment. You need systems/networking/devops basics.

So no, CS does not disappear. The emphasis shifts from manual implementation to conceptual control.

我目前的模型：AI 时代的编程三境界

我把人群分为三个层次：

  • 第一层：消费者型构建者（Consumer-builder） 这个人不会写代码，但可以通过给 AI 发提示词来做出东西。他们能做网站、Demo、脚本，或许还能做点小 App。这已经很有用了。很多人会停留在这个层次，这完全没问题。
  • 第二层：AI 辅助程序员（AI-assisted programmer） 这个人至少掌握一门语言，能看懂代码、调试错误、理解文件结构、函数、API、数据结构和基础架构。他们极度依赖 AI，但离开 AI 也不至于手足无措。这是绝大多数严肃的独立创作者的最佳状态（Sweet Spot）。
  • 第三层：工程师 / 计算机科学家（Engineer / computer scientist） 这个人对系统有着深刻的理解。他们能设计健壮的软件，推导性能、安全性、正确性、可维护性和抽象层。他们不仅让 AI 去构建，还知道如何拆解问题，以便 AI 能够安全地提供协助。

针对你的目标——教学、内容创作、数学/统计/金融/CS 的学习，或许还有构建你自己的教育工具——我认为你应当先以第二层为目标，然后在需要的地方选择性地进入第三层。

你不需要立刻成为一个传统的软件工程师。但是，拥有技术素养将会给你带来巨大的杠杆效应。

My current model: three levels of AI-era coding

I would divide people into three levels.

  • Level 1: Consumer-builder This person cannot code, but can prompt AI to make things. They can make websites, demos, scripts, maybe small apps. This is already useful. Many people will stay here, and that is fine.
  • Level 2: AI-assisted programmer This person knows at least one language, can read code, debug errors, understand files, functions, APIs, data structures, and basic architecture. They use AI heavily, but they are not helpless without it. This is the sweet spot for most serious independent creators.
  • Level 3: Engineer / computer scientist This person understands systems deeply. They can design robust software, reason about performance, security, correctness, maintainability, and abstraction. They do not merely ask AI to build. They know how to decompose a problem so that AI can help safely.

For your goals — teaching, content creation, math/statistics/finance/CS learning, maybe building your own educational tools — I think you should aim for Level 2 first, then selectively enter Level 3 where it matters.

You do not need to become a traditional software engineer immediately. But becoming technically literate will give you enormous leverage.

核心关键词：“拥有工具”（Owning the tool）

在心理上，以下两种状态有着巨大的区别：

“这是 AI 帮我做的。我希望它能一直正常运行。”

以及

“这是 AI 帮我做的。我理解它的结构。我可以修改它。”

第一种感觉很神奇，但极其脆弱。 第二种感觉很强大。

这就是现在还要学习编程的原因。 不是因为 AI 太弱。 而是因为 AI 太强了。

AI 变得越强，保持完全的盲目就越危险。一个弱小的工具帮不了你多少。而一个强大的工具可以带你走得很远——包括在错误的道路上走得极远。

学习计算机科学，就是你把双手牢牢握在方向盘上的方式。

用一句话总结我的回答 > 是的，学习编程依然有意义——不是因为你必须亲手写下每一行代码，而是因为编程知识能让你精准地指挥 AI、评判它的产出、调试它的错误，并将一次性生成的代码产物，变成你真正掌控的“系统”。

或者说得更直白残酷一点：

在过去，编程是让计算机听你话的能力。 在 AI 时代，编程是让 AI 去让计算机听你话的能力——并且在这个过程中，你不会愚弄到你自己。

The key phrase: “owning the tool”

There is a huge psychological difference between:

“AI made this for me. I hope it keeps working.”

and

“AI helped me make this. I understand the structure. I can change it.”

The first feels magical but fragile. The second feels powerful.

That is the reason to learn programming now.

Not because AI is weak. Because AI is strong.

The stronger AI becomes, the more dangerous it is to remain completely blind. A weak tool does little for you. A powerful tool can take you very far — including very far in the wrong direction.

Learning CS is how you keep your hands on the steering wheel.

My answer in one sentence

Yes, learning programming still makes sense — not because you must write every line yourself, but because programming knowledge lets you command AI precisely, judge its output, debug failures, and turn one-off generated artifacts into systems you actually own.

Or more brutally:

In the past, programming was the ability to make the computer obey you. In the AI era, programming is the ability to make the AI make the computer obey you — without fooling yourself.

第二个问题

Q: 让我们暂时把问题局限在编程语言上, 而且, 不要考虑我的工作, 我是说假设我们讨论的是一个general person.

这就是品味。 品味并不意味着精英式的审美判断。它意味着当某个东西混乱、脆弱或愚蠢时，你能直觉地感受到。 在有了 AI 之后，品味变得更加重要，因为 AI 可以极其迅速地生成海量的平庸代码。 在旧世界里，糟糕的代码受限于人类的打字速度。 在新世界里，糟糕的代码能像兔子一样疯狂繁殖。

第二层可以帮助你对 AI 说：

不。把它简化。 把这个拆分成不同的函数。 把数据放到一个单独的文件里。 只在有帮助的地方加注释。 不要重复相同的逻辑。 明确地处理错误。 删掉不必要的依赖项。

这不是计算机科学的浪漫主义。这是实用的卫生习惯。

Level 2 gives you taste

This sounds vague, but it is real.

Once you learn a little programming, you begin to notice bad structure.

You see code and think:

Why is everything in one giant file?
Why are the product names hardcoded?
Why are there five copies of the same function?
Why is the password written directly in the code?
Why does changing one color require editing fifteen places?
Why does the script silently ignore errors?

This is taste.

Taste does not mean elite aesthetic judgment. It means you can sense when something is messy, fragile, or stupid.

And with AI, taste becomes even more important, because AI can generate huge amounts of mediocre code very quickly.

In the old world, bad code was limited by human typing speed.
In the new world, bad code can reproduce like rabbits.

Level 2 helps you say:

No. Make it simpler.
Split this into functions.
Put the data in a separate file.
Add comments only where helpful.
Do not repeat the same logic.
Handle errors explicitly.
Remove unnecessary dependencies.

That is not computer science romance. That is practical hygiene.

第二层改变了你与“错误”的关系

这一点几乎是哲学层面的。 对于第一层的人来说，一个错误通常是一堵墙。 对于第二层的人来说，一个错误是一条信息。 这是一个巨大的心态转变。

当你看到： TypeError: unsupported operand type(s) for +: 'int' and 'str' 第一层看到的是：“计算机讨厌我。” 第二层看到的是：“啊，一个值是数字，另一个是文本。”

这听起来很小，但在情感上却很重要。编程变得不再那么神秘。机器变得令人烦躁，但不再神圣。 这一点之所以重要，是因为 AI 有时会加剧人的无助感。你习惯了索取、接收、粘贴。当它失败时，你会觉得自己被驱逐出了神庙。 第二层让你得以进入神庙的厨房，并看清那些所谓的神明不过是贴错了标签的变量。

Level 2 changes your relationship to error

This one is almost philosophical.

For a Level 1 person, an error is often a wall.

For a Level 2 person, an error is information.

That is a huge mental shift.

When you see:

TypeError: unsupported operand type(s) for +: 'int' and 'str'

Level 1 sees: “The computer hates me.”

Level 2 sees: “Ah, one value is a number and one is text.”

This sounds small, but emotionally it matters. Programming becomes less mystical. The machine becomes annoying, but not divine.

And that matters because AI can sometimes increase helplessness. You become used to asking, receiving, pasting. When it fails, you feel like you are outside the temple.

Level 2 lets you enter the temple kitchen and see that the gods are just badly labeled variables.

即使你从不从零手写代码，第二层依然有用

这是很多人漏掉的一点。 核心问题并不是：

我是否要亲手编写所有的代码？

大概率不需要。 核心问题是：

我能否明智地参与到编码的过程中去？

这其中包括：阅读、修改、测试、提问、审查、调试、拒绝和简化。 一个当代的“编程学习者”也许应该减少面对空白页面手写代码的时间，而把更多的时间花在这些活动上：

  • 这是 AI 生成的代码。它是干什么用的？
  • 它在什么地方可能会崩溃？
  • 我该怎么测试它？
  • 我怎样才能让它更容易被修改？
  • 我怎样才能删减它？
  • 我怎样才能让它适配我现实中的真正问题？

这就是为什么第二层对普通人来说是务实可行的。你不需要建立一个“面对白纸单枪匹马拯救世界的程序员”的身份认同。 你只需要拥有足够的素养，把代码当作一个鲜活的对象去互动。

Level 2 is useful even when you do not write code from scratch

This is the point many people miss.

The question is not:

Will I personally write all the code?

Probably not.

The question is:

Can I participate intelligently in the coding process?

That includes:

  • Reading.
  • Editing.
  • Testing.
  • Asking.
  • Reviewing.
  • Debugging.
  • Rejecting.
  • Simplifying.

A modern “programming learner” should probably spend less time writing code from a blank page and more time doing these activities:

Here is AI-generated code. What does it do?
Where could it fail?
How can I test it?
How can I make it easier to change?
How can I reduce it?
How can I adapt it to my real problem?

This is why Level 2 is realistic for ordinary people. You do not need the blank-page heroic programmer identity.

You need enough literacy to interact with code as a living object.

微小的能力会产生复利。

  • 啊，API 返回的是 JSON。

  • 啊，网页就是 HTML/CSS/JS。

  • 啊，文件路径很烦人，但可以理解。

Correct.

At Level 2, each task leaves residue in your mind:

Ah, webpages are HTML/CSS/JS.

Ah, errors have patterns.
Ah, data cleaning always takes longer than expected.
Ah, file paths are annoying but understandable.

After enough small tasks, you become much more capable without formally “studying CS.”

This is the hidden power of Level 2: transfer.

You learn something from the clothing website, and it helps with the inventory script.
You learn something from the inventory script, and it helps with sales analysis.
You learn something from sales analysis, and it helps with ad tracking.
You learn something from ad tracking, and it helps with customer segmentation.

Level 1 outputs do not compound much.
Level 2 understanding compounds.

不。绝对不是。 对于很多人来说，停留在第一层是完全理性的。 如果一个人只是偶尔需要：一个简单的网页、一个一次性的脚本、一个小小的自动化、一个玩具式的计算、或者一个纯属个人好奇的项目，那么第一层加上 AI 就完全足够了。

因此，我会把规则总结为：

这是最清晰的标准。

But is Level 2 always worth it?

No. Definitely not.

For many people, staying at Level 1 is completely rational.

If someone only occasionally needs:

A simple webpage,
a one-off script,
a small automation,
a toy calculation,
a personal curiosity project,

then Level 1 plus AI may be enough.

And this is important: we should not turn “learn programming” into moral propaganda.

Not everyone needs programming. Not everyone needs CS. Not everyone needs to “level up.” Some people have better things to do. Life is short. Attention is finite. The world is already full of people telling you to learn fifty skills before breakfast.

So I would phrase the rule like this:

Stay at Level 1 if code is only an occasional disposable tool.
Move to Level 2 if code becomes part of your repeated workflow, income, decision-making, or intellectual exploration.

That is the cleanest criterion.

My revised answer

It is because Level 2 gives five practical powers:

Second, judgment power — you can tell whether the output is plausible, fragile, or wrong.

Third, repair power — you can debug ordinary failures.

Fourth, workflow power — you can turn one-off snippets into repeatable tools.

Fifth, independence power — you are less trapped by platforms, templates, and black-box AI.

So the real value of Level 2 is not that you become someone who writes code all day.

The real value is:

You become a person who can use code as a medium of thought and action.

That, to me, is the compelling reason.

Not for everyone.

But for far more people than before.

  • 例如： 最终结果应当允许我仅通过编辑 CSV/JSON 文件来添加新产品，而不需要手动更改 HTML。

计算机科学（CS）中的提示词工程有几种不同的模式

当人们提到“提示词工程”时，他们往往把不同的活动混为一谈。对于编程，我将其划分为六种模式。

  • 模式 1：探索模式（Exploration prompt） 用于当你不知道如何着手解决问题时。

    例如： 我想建立一个基于浏览器的微型火星任务模拟。我知道物理目标，但我不确定如何设计软件结构。请给我三种可能的实现方案，从最简单到最写实。对比它们的权衡。 这是很好的前期提示词。你还没有让 AI 去构建，你是在让它绘制地形图。

  • 模式 2：规范模式（Specification prompt） 用于当你想要定义需求时。

    例如： 将以下想法转化为一份清晰的软件规范。识别出缺失的需求、边缘情况、所需的数据，以及可能的分版本（版本 1 / 版本 2）规划。 这极其有价值。AI 非常擅长帮你发现遗漏的拼图。

  • 模式 3：计划模式（Planning prompt） 在落地实现之前使用。

    例如： 根据这份规范，提出一个分成微小步骤的实现计划。每一步都应当产生一个可测试的结果。先不要写代码。 这可以防止混乱。

  • 模式 4：编码模式（Coding prompt） 用于实际的落地编写。

    例如： 仅实现第 1 步：从 products.json 加载产品数据并在产品列表页上渲染出产品卡片。先不要实现筛选功能。保持代码简单，并解释如何进行测试。 注意这句话：仅实现第 1 步。非常重要。不要让 AI 一口吞下整个项目。

  • 模式 5：调试模式（Debugging prompt） 当某些地方损坏报错时使用。

    例如： 预期的行为是：[描述] 实际的行为是：[描述] 错误信息是：[粘贴] 相关代码是：[粘贴] 请找出最可能的原因。先提出最小的修复方案。不要重写无关的部分。 这比“为什么我的代码不能用？”要好得多。

  • 模式 6：审查模式（Review prompt） 在代码写完之后使用。

    例如： 审查这段代码的正确性、安全性、可维护性和不必要的复杂性。先不要重写它。首先按照严重程度顺序列出最首要的问题。 这把 AI 变成了审查员，而不仅仅是生成器。

Used when you want to define requirements.

This is extremely valuable. AI is good at helping you discover missing pieces.

Used before implementation.

This prevents chaos.

Mode 4: Coding prompt

Used for actual implementation.

Example:

Used when something breaks.

                Here is the error message:[paste]
                Here is the relevant code:[paste]

Used after code is written.

Example:

This turns AI into a reviewer, not just a generator.

                [空输入、无效数据、海量数据、文件缺失、怪异的用户行为等]

                成功标准：
                [如何得知任务已完成]

                你的任务：
                [提出问题 / 制定计划 / 编写代码 / 审查 / 调试]

                重要提示：
                请勿 [AI 应当避免的特定事项]。
                在 [编码/更改任何内容] 之前，首先 [提出问题 / 解释计划 / 识别风险]。

这看起来很长，但原理很简单：

告诉 AI 它正在其中运行的“世界”是怎样的。

大多数糟糕的提示词之所以失败，是因为它们只描述了期望的输出，而没有描述围绕着它的世界。

For programming tasks, I would use this template:

I want to build / fix / analyze [thing].

                Context:[What is this for? Who will use it? Why does it matter?]

                Goal:[What should the final result accomplish?]

                Current state:[What already exists? Code? Files? Data? Nothing yet?]

                Requirements:
                - [Requirement 1]
                - [Requirement 2]
                - [Requirement 3]

                Non-requirements:
                - [What should NOT be included]
                - [What is out of scope for now]

                Inputs:[What data / files / user actions / parameters go in?]

                Outputs:[What should be displayed / saved / returned / generated?]

                Constraints:[Language, framework, platform, performance, style, security, simplicity, deadline, etc.]

                Edge cases:[Empty input, invalid data, large data, missing file, weird user behavior, etc.]

                Success criteria:[How we know the task is done]

                Your task:[Ask questions / make a plan / write code / review / debug]

                Important:Do not [specific thing AI should avoid].

                Before [coding/changing anything], first [ask questions / explain plan / identify risks].

This looks long, but the principle is simple:

Tell AI the world it is operating inside.

Most weak prompts fail because they only describe the desired output, not the world around it.

When talking to AI, there are two types of prompts.

Type A: Product prompt

Type B: Engineering prompt

Beginners often only give Type A prompts.

Experts give both — but not always at the same time.

First:

Then:

Then:

This keeps you from prematurely choosing the wrong technology.

即使你还不是完全的专家，这也能帮到你。 AI 变成了需求访谈员。 这是提示词工程非常具有现实意义的一个应用。

This is excellent because it uses AI to reveal the hidden structure of the problem.

This helps even if you are not yet fully expert.

AI becomes a requirements interviewer.

That is a very real use of prompt engineering.

这很漂亮。它既能防止构建不足，又能防止过度构建。

提示词工程无法脱离编程知识而独立存在

现在回到你更深层次的问题：

假设我已经对一个问题非常的了解。我所需要的只是通过 CS 实现我的想法，那么这个时候，所谓的提示词工程到底有没有价值？

有的，但它的价值取决于你的编程素养。

  • 在第一层，提示词工程主要就是：描述我想要什么。
  • 在第二层，提示词工程变成了：指定结构、约束条件、数据、行为、测试以及变动的边界。
  • 在第三层，提示词工程变成了：在一个更大的架构内部委派子问题，同时保持正确性和可维护性。

所以提示词工程和编程知识是相辅相成的。 你懂的编程越多，你的提示词就会变得越好。 你的提示词变得越好，你学习编程的速度就越快。 这是一个重要的正反馈回路。

  • 第一层的人会说：让它跑通。
  • 第二层的人会说：Bug 大概率在筛选函数里。这是预期的输入和输出。添加一个最小的测试用例，并且只修复那一个函数。
  • 第三层的人会说：这个模块违反了数据解析与渲染的分离原则。将其重构为纯数据转换函数和 UI 渲染函数。保留现有行为并添加回归测试（regression tests）。

Prompt engineering is not independent of programming knowledge

Now to your deeper question:

If I already understand the problem thoroughly and just need CS to realize it, does prompt engineering hold value?

Yes, but the value depends on your programming literacy.

At Level 1, prompt engineering is mostly:

Describe what I want.

At Level 2, prompt engineering becomes:

Specify structure, constraints, data, behavior, tests, and change boundaries.

At Level 3, prompt engineering becomes:

Delegate subproblems inside a larger architecture while preserving correctness and maintainability.

So prompt engineering and programming knowledge reinforce each other.

The more programming you know, the better your prompts become.
The better your prompts become, the faster you learn programming.

This is an important feedback loop.

A Level 1 person says:

Make it work.

A Level 2 person says:

The bug is probably in the filtering function. Here is the expected input and output. Add a minimal test case and fix only that function.

A Level 3 person says:

This module violates the separation between data parsing and rendering. Refactor it into pure data transformation functions and UI rendering functions. Preserve existing behavior and add regression tests.

我的实用法则：像给一个能力出众但健忘的助理布置工作一样去写提示词

这是最好的心智模型。 AI 不完全是员工，不完全是搜索引擎，也不完全是编译器。 对于编码任务，请把它当成：

一个速度极快、知识渊博的助理。他可以迅速地产出工作，但他可能会误解上下文、过度概括、遗漏边缘情况，并且会极其自信地做出一些值得商榷的选择。

所以你的提示词应当提供：

  • 上下文，因为他可能不了解你的处境。
  • 约束条件，因为他可能会过度构建。
  • 示例，因为示例能锚定行为。
  • 成功标准，因为“好”是一个模糊的词。
  • 边界，因为他可能会改动不相关的代码。
  • 测试，因为他可能会犯错。

这才是真正的艺术。

My practical rule: prompt like you are assigning work to a competent but forgetful assistant

This is the best mental model.

AI is not exactly an employee, not exactly a search engine, not exactly a compiler.

For coding tasks, treat it like:

A fast, knowledgeable assistant who can produce work quickly, but may misunderstand context, overgeneralize, omit edge cases, and confidently make questionable choices.

So your prompt should provide:

Context, because it may not know your situation.
Constraints, because it may overbuild.
Examples, because they anchor behavior.
Success criteria, because “good” is vague.
Boundaries, because it may change unrelated things.
Tests, because it may be wrong.

That is the real art.

一个具体的“严肃提示词”工作流

假设你有一个细节详尽的想法。我不会从实现代码开始。我会使用这个序列：

  • 提示词 1：需求发掘

    我想构建 [项目]。这是我的粗略想法：[想法]。先不要提供代码。首先，提出你认为将此想法转化为精确的软件规范所需的最重要的问题。将问题按类别分组。同时也请指出我的想法中包含哪些隐藏的假设。

  • 提示词 2：制定规范 （在回答完问题后）：

    根据我的回答，撰写一份简洁的软件规范。包含目标、用户、核心功能、非功能定义、数据模型、用户流程、边缘情况和成功标准。保持版本 1 务实可行。

  • 提示词 3：架构/方案选择

    鉴于这份规范，提出 2–3 种技术方案。从简洁性、可维护性、性能、成本和难度方面进行对比。为版本 1 推荐一种方案。

  • 提示词 4：实现计划

    使用推荐的方案，将落地实现拆分为微小的步骤。每一步应当能够独立测试。先不要写代码。

  • 提示词 5：编写一步代码

    仅实现第 1 步。保持代码简洁。解释如何运行和测试它。不要实现后续的步骤。

  • 提示词 6：审查

    审查这一步的实现，看其在正确性、简洁性、可维护性和边缘情况方面表现如何。在重写任何内容之前，先列出问题。

这是严谨规训过的方式。 而糟糕的方式是：把所有东西建出来。那只会产出一堆代码。有时候可能是一堆好看的代码，但它依然只是一堆代码。

A concrete “serious prompt” workflow

Suppose you have a detailed idea. I would not begin with implementation. I would use this sequence:

Prompt 1: Requirements discovery

I want to build [project]. Here is my rough idea:[idea]Do not propose code yet. First, ask me the most important questions needed to turn this into a precise software specification. Group the questions by category. Also point out any hidden assumptions in my idea.

Prompt 2: Specification

After answering questions:

Based on my answers, write a concise software specification. Include goal, users, core features, non-features, data model, user flow, edge cases, and success criteria. Keep Version 1 realistic.

Prompt 3: Architecture/options

Given this specification, propose 2–3 technical approaches. Compare them by simplicity, maintainability, performance, cost, and difficulty. Recommend one approach for Version 1.

Prompt 4: Implementation plan

Using the recommended approach, break the implementation into small steps. Each step should be independently testable. Do not write code yet.

Prompt 5: Code one step

Implement Step 1 only. Keep the code simple. Explain how to run and test it. Do not implement later steps.

Prompt 6: Review

Review the implementation for correctness, simplicity, maintainability, and edge cases. List problems before rewriting anything.

This is the disciplined way.

The worse way is:

Build everything.

That produces a pile. Sometimes a beautiful pile, but still a pile.

最深的一点：提示词工程是思维的外化

在旧的编程中，你写代码是为了强迫计算机服从。 在 AI 辅助编程中，你写提示词是为了强迫你自己的意图变得足够清晰，以便 AI 能够提供协助。 所以提示词工程之所以有价值，并不是因为 AI 需要魔法般的字眼。 它有价值，是因为我们人类是模糊的生物。 我们以为自己知道自己想要什么。然后我们试图去告诉 AI，突然之间，迷雾就出现了：

  • 库存为零时到底应该发生什么？
  • 用户应该在线付款还是发送询盘？
  • 模拟器可接受的最低精确度是多少？
  • 火星模型应该是物理上写实的，还是视觉上具有教育意义的？
  • 代码应当优先考虑简洁性还是性能？
  • “好看的网站”是什么意思？
  • “模拟”是什么意思？

撰写提示词的过程逼着你暴露出这些隐藏的决策。 这就是为什么我不会贬低提示词工程的原因。浅显版的提示词工程是荒谬的，而严肃版的提示词工程本质上是一种将意图转化为可操作结构的科学。

The deepest point: prompt engineering is thinking made external

In old programming, you wrote code to force the computer to obey.

In AI-assisted programming, you write prompts to force your own intention to become clear enough that AI can help.

So prompt engineering has value not because AI needs magic phrasing.

It has value because we are vague creatures.

We think we know what we want. Then we try to tell AI, and suddenly the fog appears:

What exactly should happen when inventory is zero?
Should users pay online or send inquiries?
What is the minimum acceptable accuracy of the simulation?
Should the Mars model be physically realistic or visually educational?
Should the code prioritize simplicity or performance?
What does “beautiful website” mean?
What does “simulate” mean?

The prompt-writing process exposes these hidden decisions.

That is why I would not dismiss prompt engineering. The shallow version is ridiculous. The serious version is basically the discipline of making intention operational.

哪些内容属于“共识性的基础”？

我们以 Python 为例。 核心基础不应该是“Python 的全部”，它应当是能够让你阅读并修改普通 Python 代码的最小语言机制。 我把它划分为十个版块。

版块 1：运行代码（Running code）

这听起来太基础了，但它不可或缺。 你应当知道：

  • 什么是 .py 文件？
  • 什么是 Notebook？
  • 如何运行一个文件？
  • 如何停止一个正在运行的程序？
  • 什么是工作目录（working directory）？
  • 什么是虚拟环境（virtual environment），至少大致了解？
  • 如何安装一个第三方包（package）？
  • 如何阅读一条报错信息？

这不光鲜。但没有它，其他的一切都会变得摇摇欲坠。一个理解函数、却无法独立运行一个文件的学习者，在实际中依然是无法开展工作的。

版块 2：值和类型（Values and types）

你需要掌握基础的对象类型：

  • 数字：int, float
  • 文本：str
  • 布尔值：bool
  • 空值：None
  • 容器：list, tuple, dict, set

但你不需要背诵所有的操作方法。 以字符串（Strings）为例，你不需要背诵整个字符串方法宇宙。 你只需要掌握这个核心概念：

  • 字符串是一个字符序列。
  • 它可以被索引（indexed）和切片（sliced）。
  • 它拥有自己的方法。
  • 某些方法会返回全新的字符串，而不是改变原来的字符串。
  • 常见的操作包括分割（splitting）、连接（joining）、替换（replacing）、去除首尾空白（stripping）、搜索（searching）以及格式化（formatting）。

这些就足够了。 你不需要死记硬背这个方法到底是叫 startswith、start_with 还是 startsWith。AI 和文档可以告诉你。但你应当知道，这类方法大概率是存在的。 这就是“死记硬背”与“方向感”的区别。

版块 3：控制流（Control flow）

你需要：

  • if / elif / else
  • for 循环
  • while 循环
  • break / continue
  • 基础的条件对比逻辑
  • 布尔值组合：and, or, not

这是最根本的，因为编程在本质上就是“重复的、带有条件判断的动作”。

版块 4：函数（Functions）

这是真正的核心之一。 你需要：

  • 如何定义一个函数
  • 参数（Parameters）与返回值（return values）
  • 默认参数
  • 局部变量（Local variables）
  • 纯函数（Pure functions）与带有副作用的函数（functions with side effects）的区别
  • 如何将重复的逻辑拆解提炼为函数

对于 AI 时代的编程，函数的重要性更加凸显，因为它们为 AI 提供了天然的、可用于审查、修改、测试和解释的独立单元。 一个巨大的、写成一坨的脚本是极难被人类监督的。 而微小的函数则非常容易被监督。

这里不是指高深的算法，而是实用的数据结构。

  • 列表（Lists）用于处理序列。
  • 字典（Dictionaries）用于处理键值对记录。
  • 集合（Sets）用于处理唯一性/成员归属。
  • 嵌套结构（Nested structures）用于处理现实中的真实数据。

例如：

你需要读取和写入东西。

  • 文本文件
  • CSV 文件
  • JSON 文件
  • 稍后可能会根据目标接触 Excel 文件

  • 保存实验或运行结果

版块 7：错误与异常（Errors and exceptions）

你需要明白，报错并不是神明对你的惩罚。 常见的类别包括：

  • Syntax error: 语法错了。
  • Name error: 某个东西没有被定义。
  • Type error: 对象的类型用错了。
  • Value error: 传入了错误的值。
  • Index error: 列表位置/索引不存在。
  • Key error: 字典里的键不存在。
  • File not found: 路径出了问题。
  • Import error: 包或环境配置出了问题。

你不需要背诵每一种错误，但你需要知道如何阅读 Traceback（报错追踪信息）。这应当放在最基础的教学里，而不是最后才被当成附带提及的琐事。

版块 8：模块与包（Modules and packages）

你需要知道：

  • 如何导入（import）东西
  • 标准库（standard library）与第三方库（third-party library）的区别
  • 如何安装一个包
  • 如何阅读一个包的基础使用示例
  • 如何避免把自己的文件命名为 requests.py 从而召唤出幽灵和 Bug

同样，这不光鲜，但极其务实。

版块 9：基础调试（Basic debugging）

不一定非要用高级的调试工具，但至少要学会：

  • 打印中间值（Print intermediate values）
  • 使用微小的测试输入进行验证
  • 隔离出导致失败的那一行代码
  • 检查变量的类型（types）和形状/维度（shapes）
  • 对比预期结果与实际结果
  • 利用 AI 来解释某一个报错
  • 避免毕其功于一役、一次性改动五个地方

这应当从一开始就作为习惯被培养。

版块 10：阅读代码（Reading code）

这一点被极其诡异地忽视了。 学习者应当练习阅读一小段代码，并回答：

  • 输入是什么？
  • 输出是什么？
  • 什么状态发生了改变？
  • 它在什么地方可能会崩溃？
  • 它做出了哪些潜在的假设？

在 AI 时代，阅读代码可能比从零开始手写代码还要重要。 因为 AI 可以写，但你必须得会读。

What belongs in the consensus-based foundation?

Let’s use Python as the example.

The core foundation should not be “all of Python.” It should be the minimum language machinery that lets you read and modify ordinary Python code.

I would divide it into ten blocks.

Block 1: Running code

This sounds too basic, but it is not optional.

You should know:

What is a .py file?
What is a notebook?
How do you run a file?
How do you stop a running program?
What is a working directory?
What is a virtual environment, at least roughly?
How do you install a package?
How do you read an error message?

This is not glamorous. But without it, everything else becomes fragile.

A person who understands functions but cannot run a file independently is still not operational.

Block 2: Values and types

You need the basic object types:

Numbers: int, float
Text: str
Truth values: bool
Nothingness: None
Containers: list, tuple, dict, set

But you do not need to memorize every method.

For strings, for example, you do not need to memorize the entire string method universe.

You need the idea:

A string is a sequence of characters.
It can be indexed and sliced.
It has methods.
Some methods return new strings instead of changing the original.
Common operations include splitting, joining, replacing, stripping, searching, formatting.

That is enough.

You do not need to memorize whether the method is called startswith, start_with, or startsWith. AI/documentation can help. But you should know that such a method probably exists.

That is the difference between memory and orientation.

Block 3: Control flow

You need:

if / elif / else
for loops
while loops
break / continue
Basic comparison logic
Boolean combinations: and, or, not

This is fundamental because programming is repeated conditional action.

Block 4: Functions

This is one of the true cores.

You need:

How to define a function
Parameters and return values
Default parameters
Local variables
Pure functions versus functions with side effects
How to decompose repeated logic into functions

For AI-era programming, functions matter even more because they create natural units for AI to inspect, modify, test, and explain.

A giant script is hard to supervise.
Small functions are easy to supervise.

Block 5: Data structures

Not advanced algorithms. Just practical structure.

Lists for sequences.
Dictionaries for key-value records.
Sets for uniqueness/membership.
Nested structures for real data.

For example:

product = {
                    "id": "A001",
                    "name": "linen shirt",
                    "sizes": ["S", "M", "L"],
                    "price": 29.99}

This kind of structure is everywhere. JSON, APIs, configuration files, scraped data, product catalogs, simulation parameters — all of them reduce to this kind of thinking.

Block 6: Input/output

You need to read and write things.

Text files
CSV files
JSON files
Maybe Excel later, depending on goal
Printing and logging
Saving results

A beginner who can only manipulate values inside a tutorial is still trapped. Real work begins when code touches files.

Block 7: Errors and exceptions

You need to know that errors are not divine punishment.

Common categories:

Syntax error: the grammar is wrong.
Name error: something is not defined.
Type error: wrong kind of object.
Value error: wrong value.
Index error: position does not exist.
Key error: dictionary key does not exist.
File not found: path problem.
Import error: package/environment problem.

You do not need to memorize every error. But you need to know how to read the traceback.

This belongs in the foundation, not as an afterthought.

Block 8: Modules and packages

You need to know:

How to import something
Difference between standard library and third-party library
How to install a package
How to read a package’s basic usage example
How to avoid naming your own file requests.py and summoning demons

Again, not glamorous. Extremely practical.

Block 9: Basic debugging

Not advanced debugging tools necessarily. But at least:

Print intermediate values
Use small test inputs
Isolate the failing line
Check types and shapes
Compare expected vs actual result
Use AI to explain an error
Avoid changing five things at once

This should be taught from the beginning.

Block 10: Reading code

This is strangely neglected.

A learner should practice reading small pieces of code and answering:

What are the inputs?
What are the outputs?
What state changes?
Where could it fail?
What assumptions does it make?

In the AI era, reading code may be more important than writing from scratch.

Because AI can write. But you must read.

4. 哪些内容应当被排除在核心工具箱之外？

这一点同样重要。 以 Python 为例，我不会把这些内容放进面向普罗大众的通用核心里：

  • 深入的面向对象编程（OOP in depth）
  • 装饰器（Decorators）
  • 深入的生成器（Generators in depth）
  • 元类（Metaclasses）
  • 并发/异步编程（Concurrency / async）
  • 深入的正则表达式（Regular expressions in depth）
  • 高级的打包部署
  • 深入的类型提示（Type hints in depth）
  • 性能优化
  • 设计模式
  • 深入的测试框架
  • Web 框架
  • GUI（图形用户界面）框架
  • 机器学习库
  • 爬虫类库
  • 数据库操作类库

这些有用吗？非常有用。 但它们是通用的基础吗？并不是。它们属于“目的箱”或“扩张箱”。 初学者应该知道这些东西的存在，但完全不需要立刻去学它们。 因此，基础阶段应当包含“认知”，而不是“精通”。 例如：

“Python 有‘类（classes）’的概念，但你在最开始不需要去深入钻研面向对象编程。” “Python 有正则表达式，对于匹配文本模式很有用，等你需要的时候再去学。” “Python 有异步（async），对处理很多网络任务很有用，但一开始可以先忽略它。” “Python 有 NumPy/Pandas，是数据工作的核心，但它们不属于语言本身的核心语法。”

这保护了学习者，不至于让他们被知识淹死。

What should be excluded from the core case?

This is just as important.

For Python, I would not put these in the universal core:

Object-oriented programming in depth
Decorators
Generators in depth
Metaclasses
Concurrency / async
Regular expressions in depth
Advanced packaging
Type hints in depth
Performance optimization
Design patterns
Testing frameworks in depth
Web frameworks
GUI frameworks
Machine learning libraries
Scraping libraries
Database libraries

Are these useful? Yes.

Are they universally foundational? No.

They belong to the goal-oriented case or expansion case.

The beginner should know these things exist, but not necessarily learn them immediately.

So the foundation should include awareness, not mastery.

Example:

“Python has classes, but you do not need deep OOP at the beginning.”
“Python has regular expressions, useful for text patterns, but learn them when needed.”
“Python has async, useful for many network tasks, but ignore it first.”
“Python has NumPy/Pandas, essential for data work, but not part of the language core.”

This protects the learner from drowning.

目标导向的基础箱（The goal-oriented foundation）

现在我们开始追问：

你需要用 Python 来干什么？

不同的目标，决定了第二层基础箱中装载不同的工具。 例如：

  • 如果目标是数据分析： 目的箱里应该是：pandas、numpy 基础、读取 CSV/Excel、清洗缺失值、分组/筛选/排序、使用 matplotlib 绘图、基础的统计工作流、导出结果。 你不需要 Flask，你一开始不需要网络爬虫，你也不需要深入学习“类”。

  • 如果目标是自动化： 目的箱里应该是：文件路径、文件夹操作、批量重命名、CSV/Excel 读写、正则表达式基础、可能包括发送电子邮件、操作 API、定时运行脚本、日志记录。 你不需要机器学习。

  • 如果目标是网络爬虫： 目的箱里应该是：HTTP 基础、HTML 基础、CSS 选择器、requests、BeautifulSoup、浏览器开发者工具基础、翻页处理、频率限制（rate limits）、保存爬取的数据、法律与道德合规考量、稍后可能会接触 Selenium/Playwright。 你不需要高深的数值计算。

  • 如果目标是科学计算： 目的箱里应该是：numpy 数组、函数与参数、图表绘制、数值误差、模拟循环、随机性、数据保存、可能需要 scipy、可能需要利用 sympy 进行符号数学计算。 你不需要 Web 框架。

  • 如果目标是 Web 开发（专门针对 Python）： 根据项目规模选择 Flask/FastAPI/Django、HTTP 请求与响应、模板渲染、路由（routing）、表单、数据库、迟早要学的身份验证、迟早要学的部署上线。 但如果目标是前端网页，Python 甚至可能都不是主语言，JavaScript/TypeScript 会更合适。

所以，你是完全正确的：对于非计算机专业的人来说，编程语言的学习必须是目标导向的。否则，它就会变成一场令人疲惫的博物馆一日游。

“这是城堡里的每一个房间。” “大兄弟，我其实只需要进厨房。”

“Here is every room in the palace.”
“Sir, I only needed the kitchen.”

但目标导向的学习存在一个危险

这个危险就是变成一个机械的“菜谱照抄者（recipe-follower）”。 例如：

“我学会了怎么用 BeautifulSoup 去爬某一个特定的网站。”

但随后：

  • 网站改版了。
  • HTML 的结构变了。
  • 页面变成通过 JavaScript 异步加载了。
  • 请求被拦截屏蔽了。
  • 数据需要翻页。
  • 编码格式奇奇怪怪。
  • 结果里包含了大量重复项。
  • 脚本只能跑通一次，无法重复运行。

这时候，你的菜谱就失效了。

这就是为什么目标导向的学习必须建立在核心箱之上的原因。 学习者不应该仅仅只知道这一行代码：

Python

soup.find_all("a")

他们应当理解：

  • 这是一个函数/方法的调用。
  • 它返回一个类似于列表的集合。
  • 其中的每一个元素都是一个代表 HTML 标签的对象。
  • 我可以去审查它。
  • 我可以用循环去遍历它。
  • 我可以提取出里面的属性
  • 我能处理缺失值。
  • 我能保存结果。

这样一来，当库发生变动或者任务改变时，他们才不至于束手无策。 所以，它们之间的关系是：

核心箱赋予你“迁移能力”。 目的箱赋予你“实用价值”。 扩张箱赋予你“成长空间”。

这三者缺一不可。

But goal-oriented learning has a danger

The danger is becoming a recipe-follower.

For example:

“I learned how to scrape one website with BeautifulSoup.”

But then:

The website changes.
The HTML structure is different.
The page loads with JavaScript.
The request is blocked.
The data is paginated.
The encoding is weird.
The result contains duplicates.
The script works once but not repeatedly.

Now the recipe breaks.

This is why goal-oriented learning must be built on top of the core case.

The learner should not merely know:

soup.find_all("a")

They should understand:

This is a function/method call.
It returns a list-like collection.
Each element is an object representing an HTML tag.
I can inspect it.
I can loop over it.
I can extract attributes.
I can handle missing values.
I can save results.

Then when the library changes or the task changes, they are not helpless.

So the relationship is:

Core case gives transfer.
Goal case gives usefulness.
Expansion case gives growth.

All three are needed.

在 AI 时代如何学习和阅读文档

这是一个极漂亮的问题。因为是的，文档通常极其冗长、乏味、写得参差不齐，有时甚至默认你早就知道了答案。 在过去，人们会说：

去读文档（Read the docs）。

这条建议既正确，又毫无用处。这就像对一个试图解决税务问题的人说“去读法律条文”一样。 在 AI 时代，我们不应该在线性地去“读文档”。我们应当去“审问”它。 我建议采用一个“五步通读法”。

第一步：询问该库解决了什么问题

在阅读任何细节之前，先发问：

这个库是干什么用的？它交回给我的核心对象（object）是什么？最常用的 5 个操作是什么？

针对 BeautifulSoup：

  • 它是用来解析 HTML/XML 的。
  • 它把原始的 HTML 文本变成了一棵可以遍历的树。
  • 你可以查找标签。
  • 你可以挑选元素。
  • 你可以提取文本和属性。
  • 如果需要，你可以修改这棵树。

这就是你的心理地图。

第二步：寻找“最小可行示例”

不要一上来就去读完整的 API 核心参考手册。去找那个最小的、能跑通的示例：

Python

from bs4 import BeautifulSoup
                
                html = "<html><body><h1>Hello</h1></body></html>"
                soup = BeautifulSoup(html, "html.parser")
                print(soup.h1.text)

这能让你摸到它的“心跳”。任何一个库，首先都应当被浓缩为一个最简心跳示例。

第三步：建立一个“常用操作”备忘单（Cheat Sheet）

针对 BeautifulSoup，也许是：

  • 解析 HTML
  • 查找第一个匹配的标签
  • 查找所有匹配的标签
  • 使用 CSS 选择器
  • 获取文本
  • 获取属性
  • 处理缺失元素
  • 循环遍历结果
  • 提取表格/列表
  • 保存数据

这个备忘单应当极其微小，不需要追求面面俱到。目的不是去概括文档，而是为了打造一扇走进去的门。

第四步：将 AI 视为文档的“翻译官”

你可以这样写提示词：

我正在阅读 BeautifulSoup 的文档。请用最精简的示例向我解释 find、find_all 和 select 之间的区别。并告诉我通常在什么情况下更倾向于使用哪一个。

或者：

这是文档中的一个段落：[粘贴]。请用大白话帮我重写它，并给我提供一个可运行的示例。

这是 AI 时代极佳的使用方式。AI 不应该完全取代官方文档，因为 AI 会产生幻觉。但 AI 可以让文档变得极易消化。 合理的工作流应当是：

官方文档是“黄金标准/真相之源”；AI 是“翻译官/口译员”；写出的代码是“试金石”。

这个铁三角结构是安全的。

第五步：做一个“微型个人项目（micro-project）”

不要为了“学 BeautifulSoup”而学。去完成一件具体的事情。 例如：

从一个静态网页中爬取文章标题，并保存为 CSV 文件。

然后进行自然延伸：

  • 顺便把链接也爬下来。
  • 顺便把日期也爬下来。
  • 顺便处理一下某些文章缺少日期的情况。
  • 顺便让它能跳转到第 2 页。
  • 顺便清洗一下空白字符。
  • 顺便过滤掉重复项。

文档就是这样在手中活过来的。

How to learn documentation in the AI era

This is a beautiful question. Because yes, documentation is often long, tedious, unevenly written, and sometimes assumes you already know the answer.

In the past, people said:

Read the docs.

This advice is both correct and useless. It is like saying “read the law” to someone trying to solve a tax problem.

In the AI era, we should not “read documentation” linearly. We should interrogate it.

I suggest a five-pass method.

Pass 1: Ask what problem the library solves

Before reading details, ask:

What is this library for? What kind of object does it give me? What are the 5 most common operations?

For BeautifulSoup:

It parses HTML/XML.
It turns raw HTML text into a navigable tree.
You can find tags.
You can select elements.
You can extract text and attributes.
You can modify the tree if needed.

That is the map.

Pass 2: Find the minimum working example

Do not start with the full API reference.

Find the smallest example:

from bs4 import BeautifulSoup
                html = "<html><body><h1>Hello</h1></body></html>"
                soup = BeautifulSoup(html, "html.parser")
                print(soup.h1.text)

This gives you the heartbeat.

Every library should first be reduced to a heartbeat example.

Pass 3: Build a “common operations” cheat sheet

For BeautifulSoup, maybe:

Parse HTML
Find first matching tag
Find all matching tags
Use CSS selector
Get text
Get attribute
Handle missing element
Loop over results
Extract table/list
Save data

This cheat sheet should be tiny. Not complete.

The goal is not to summarize the documentation. The goal is to create an entry door.

Pass 4: Use AI as a documentation translator

Prompt:

I am reading the BeautifulSoup documentation. Explain the difference between find, find_all, and select with minimal examples. Also tell me when each one is usually preferred.

Or:

Here is a documentation paragraph. Rewrite it in plain English and give me one runnable example.

This is an excellent AI-era use. AI should not replace docs entirely, because it can hallucinate. But AI can make docs digestible.

The workflow should be:

Docs are the source of truth.
AI is the interpreter.
Code is the test.

That triangle is safe.

Pass 5: Make one micro-project

Do not “learn BeautifulSoup.” Do one concrete thing.

Example:

Scrape article titles from one static webpage and save them to CSV.

Then extend:

Also scrape links.
Also scrape dates.
Also handle missing dates.
Also go to page 2.
Also clean whitespace.
Also avoid duplicates.

That is how documentation becomes alive.

如何避免觉得“每一个新库都是一个完全陌生的世界”

这就是“扩张箱”发挥作用的地方了。 绝大多数库在底层都面临着相同的、需要被解答的学习问题：

  • 它解决了什么问题？
  • 它的核心对象（central object）是什么？
  • 我如何创建出这个对象？
  • 常见有哪些操作方法？
  • 它预期接收什么样的输入？
  • 它会返回什么样的输出？
  • 常见有哪些错误报错？
  • 最小的、能运行的示例是什么？
  • 我该如何审查和窥探中间环节的运行结果？
  • 我该如何保存结果，或者把它传给下一步？

这是一个普适通用的模板。

  • 对于 BeautifulSoup，核心对象是：soup、标签对象（tags）、结果集。
  • 对于 pandas，核心对象是：DataFrame, Series。
  • 对于 matplotlib，核心对象是：figure/axes、画布、图表对象。
  • 对于 requests，核心对象是：响应对象（response）。
  • 对于 numpy，核心对象是：多维数组（array）。
  • 对于 Flask/FastAPI，核心对象是：app、路由、请求/响应。

一旦你学会了去问这套相同的问题，新的第三方库在你面前就会停止展现那种完全的陌生感。它们依然需要你付出努力，但其轮廓已经变得熟门熟路。

这完全契合你关于做菜的比喻：一旦你掌握了那些底层的大问题——火候、油脂、酸度、盐度、质感、时机——一道新菜对你来说固然是新的，但绝不再会是一场宇宙级的恐怖灾难。

How to avoid feeling like every new library is a strange world

This is where the expansion case comes in.

Most libraries have the same underlying learning questions:

  1. What problem does it solve?
  2. What is the central object?
  3. How do I create that object?
  4. What are the common operations?
  5. What input does it expect?
  6. What output does it return?
  7. What errors are common?
  8. What is the smallest working example?
  9. How do I inspect intermediate results?
  10. How do I save or pass results to the next step?

This is the universal template.

For BeautifulSoup, central object: soup, tags, result sets.

For pandas, central object: DataFrame, Series.

For matplotlib, central object: figure/axes, plots.

For requests, central object: response.

For numpy, central object: array.

For Flask/FastAPI, central object: app, routes, request/response.

Once you learn to ask these same questions, new libraries stop feeling completely alien.

They still require effort. But the shape is familiar.

This is like your cooking analogy: once you know the general questions — heat, fat, acid, salt, texture, timing — a new dish is still new, but not cosmic horror.

全新的反馈回路

现在谈谈关于记忆和记住的问题。 你是对的：旧的反馈回路通常是：

记笔记 ➔ 做习题 ➔ 也许复习笔记 ➔ 反正最后还是忘光。

而且许多练习题对于已经离开校园的、成年自学者来说，确实有一种被羞辱智商的感觉。回文字符串检测就是最经典的罪魁祸首。它本身没有罪，但确实，很多学习者会本能地发出灵魂拷问：“我做这个到底是为了图啥……”

在 AI 时代，我认为反馈回路可以被重新设计为以下模型：

实际需求（Need） ➔ 动手构建（Build） ➔ 搞砸折腾（Break） ➔ 解释复盘（Explain） ➔ 修复改进（Repair） ➔ 迁移复用（Reuse） ➔ 提炼压缩（Compress）

让我逐一解释。

  • Step 1: 实际需求（Need） 从一个真实或半真实的需要开始。不一定非得是一个宏大的项目，但必须有一个“师出有名”的理由。

    • 糟糕： 通过检测回文字符串来练习字符串。

    • 更好： 清洗一份杂乱的产品名称列表；规范化某个文件夹里的文件名；从一堆乱七八糟的文本里提取出日期；格式化学生的成绩；解析网页上的标题；把散落的笔记整理成表格；批量重命名文件夹里的图片。真实的需要能带来情绪上的摩擦力。你是在动真格的，你会在意它。

  • Step 2: 动手构建（Build） 借助 AI、官方文档和你自己的思考，先把第一版捏出来。不需要追求纯洁和完美，重点是先让它动起来。

  • Step 3: 搞砸折腾（Break） 故意去测试一些奇奇怪怪的情况。如果输入是空的呢？如果文件丢了呢？如果文本里塞满了多余的空格呢？如果里面中英文混杂呢？如果数字被当成字符串存储了呢？如果网页没有标题呢？如果某件产品没有写价格呢？把东西折腾坏，才是真正的学习。

  • Step 4: 解释复盘（Explain） 在代码能够正常运转后，对 AI 发问：

    帮我逐行解释这段代码。解释一下里面的数据流。它做出了哪些潜在的假设？哪一部分是最脆弱、最容易出 Bug 的？ 这是 AI 极其擅长且极具实用价值的环节。

  • Step 5: 修复改进（Repair） 针对暴露出的一个弱点进行修复。不要推倒重来，只做一项局部的改进。也许是加上错误处理，也许是拆分成函数，也许是把变量名改得更清晰，也许是把输出保存下来。

  • Step 6: 迁移复用（Reuse） 在另一个高度相似、临近的任务中再次复用这一套模式。这就是记忆形成的过程。 例如：任务 1 用来清洗产品名称 ➔ 任务 2 用来清洗学生姓名 ➔ 任务 3 用来清洗下载的文件名 ➔ 任务 4 用来清洗爬取到的文章标题。现在，字符串操作方法对你来说不再是抽象的符号，它们成了你手里用顺了的工具。

  • Step 7: 提炼压缩（Compress） 只有在积累了实战经验之后，你才去写笔记。 绝不要在实战之前写。笔记应当是战斗结束后的“战利品浓缩物”。 例如：

    字符串清洗模式：1. 去除首尾空白 2. 统一大小写 3. 替换掉不需要的字符 4. 必要时进行分割 5. 必要时进行合并 6. 测试空值/缺失值的情况。 这种笔记之所以有用，因为它是从真实的战火中提炼出来的。

The new feedback loop

Now the memory problem.

You are right: the old loop was usually:

take notes → do exercises → maybe review notes → forget anyway

And many exercises feel insulting to adult self-directed learners. Palindrome detection is the classic offender. It is not evil, but yes, many learners will naturally ask: “Why am I doing this?”

In the AI era, I think the feedback loop should be redesigned as:

need → build → break → explain → repair → reuse → compress

Let me explain.

Step 1: Need

Start from a real or semi-real need.

Not necessarily a grand project. But something with a reason.

Bad:

Practice strings by detecting palindromes.

Better:

Clean a list of product names.
Standardize file names.
Extract dates from messy text.
Format student scores.
Parse titles from webpages.
Convert notes into a table.
Rename image files in a folder.

A real need gives emotional friction. You care.

Step 2: Build

Use AI, documentation, and your own thinking to build the first version.

The point is not to be pure. The point is to get moving.

Step 3: Break

Deliberately test weird cases.

What if input is empty?
What if the file is missing?
What if text has extra spaces?
What if there is Chinese and English mixed?
What if numbers are stored as strings?
What if the webpage has no title?
What if one product has no price?

Breaking is learning.

Step 4: Explain

After the code works, ask:

Explain this code line by line.
Explain the data flow.
What are the assumptions?
Which part is most fragile?

This is where AI is very useful.

Step 5: Repair

Fix one weakness.

Do not rewrite everything. Add one improvement.

Maybe add error handling.
Maybe split into functions.
Maybe rename variables.
Maybe save output.
Maybe handle missing data.

Step 6: Reuse

Use the same pattern in another nearby task.

This is how memory forms.

For example:

Task 1: clean product names.
Task 2: clean student names.
Task 3: clean downloaded filenames.
Task 4: clean scraped article titles.

Now string methods are no longer abstract. They become tools.

Step 7: Compress

Only after experience do you write notes.

Not before.

The note should be a compressed artifact of use.

Example:

String cleaning pattern:
                1. strip whitespace
                2. normalize case
                3. replace unwanted characters
                4. split if needed
                5. join if needed
                6. test empty/missing values

This note is useful because it came from battle.

笔记应该扮演什么角色？

对于编程，我建议只记录四种类型的笔记：

  • 类型 1：概念笔记（Concept notes） —— 对想法的极简解释。

  • 类型 2：模式笔记（Pattern notes） —— 可复用的微型解决方案。

    读取 CSV ➔ 清洗行数据 ➔ 进行数据转换 ➔ 保存为新的 CSV。

  • 类型 3：错误笔记（Error notes） —— 个人踩坑日记（Bug 日记）。

    FileNotFoundError: 通常是因为我在错误的文件夹下运行了脚本。检查 os.getcwd() 或者使用绝对路径。 这是真正的金子。因为你未来会踩的 Bug，往往只是你以前踩过的 Bug 的全息复刻版。

  • 类型 4：项目笔记（Project notes） —— 如何去运行、修改和维护一个你亲手建好的东西。

    要更新产品展示网站，只需编辑 products.json，把图片放进 /images 文件夹，然后运行 python build.py。 这其实比传统的课本式学习笔记要有用得多。

更好的笔记长这样：

“在清洗爬取到的文本时，这是我一直在反复使用的 5 个操作。” 这样的笔记才活着。

What role should notes play?

Your question about notes is sharp: do people really reread notes from years ago?

Usually no.

So perhaps the primary role of notes is not long-term storage. The primary role is thinking during compression.

Notes are not a warehouse. Notes are a refinery.

When you write a good note, you transform messy experience into reusable structure.

For programming, I would keep four types of notes.

Type 1: Concept notes

Short explanations of ideas.

Example:

A dictionary represents mapping: key → value. Use it when lookup by name/id is more natural than position.

Type 2: Pattern notes

Reusable mini-solutions.

Example:

Type 3: Error notes

Personal bug diary.

Example:

This is gold. Because your future bugs are often recycled versions of your old bugs.

How to run, modify, and maintain a thing you built.

Example:

This is actually more useful than traditional study notes.

That becomes dead.

Better notes are:

That lives.

编程中的“理解”与物理学中的“理解”是不同的

你做出了一个非常重要的区分：在物理学中，记忆通常源于真正的、底层的公式推导和理解；而在编程中，记忆往往来自于反复的使用。 我非常赞同，但我还想补充一点：编程包含三种不同维度的“理解”。

  1. 概念上的理解（Conceptual understanding）

    • 例如： 函数是一个将输入转化为输出的、可重复使用的黑盒。

  2. 操作上的理解（Operational understanding）

    • 例如： 我能在 Python 中正确地写出并调用一个函数。

  3. 情境上的理解（Situational understanding）

    • 例如： 我明确知道在什么时候，把这一坨代码提炼成一个独立的函数是完全值得的。

大多数教程都把所有精力焊死在了“操作上的理解”上。而真正优秀的学习应当将这三者完美结合。

以列表（Lists）为例：

  • 概念上： 列表是一个有序的集合。

  • 操作上： 我会使用索引、切片、append、循环、排序。

  • 情境上： 当顺序和重复至关重要时，我用列表；当唯一性是核心时，我用集合（set）；当通过名称检索是核心时，我用字典（dictionary）。

第三个层面（情境层面）极其重要。它才是让你具备“设计能力”的关键。 操作层面的细节可以全权交由 AI 协助，但情境层面的大局观判断必须依赖你自己的实战。

You made an important distinction: in physics, memory often comes from true understanding. In programming, memory often comes from repeated use.

Conceptual understanding

Example:

A function is a reusable transformation from inputs to outputs.

Operational understanding

Example:

I can write and call a function correctly in Python.

Situational understanding

Example:

I know when it is worth turning code into a function.

  • 每一个第三方包的庞大 API。
  • 语法的每一个极端冷门边缘用例。

但你应当具备足够的知识，多到在看到一个问题时能够瞬间识别出：

“能够识别（Recognition）”比“能够盲写（Recall）”要重要得多。 这是一个颠覆性的转变。

What should we memorize?

In the AI era, memorize less syntax, but memorize more structure.

You should memorize:

The basic control structures.
The main data types.
How functions work.
How to run code.
How to read errors.
The idea of input/output.
The idea of data transformation.
The common shapes of problems.

You do not need to memorize:

Every string method.
Every list method.
Every argument of every library function.
Every package’s API.
Every syntax corner case.

But you should know enough to recognize:

“Ah, this is probably a string operation.”
“This is probably a dictionary lookup.”
“This is probably a file path problem.”
“This is probably a loop over records.”
“This is probably a parsing problem.”
“This is probably a missing package problem.”

Recognition is more important than recall.

That is a huge shift.

最深层的颠覆

在 AI 出现之前，学习一门编程语言的路径通常是：

学习语法 ➔ 做练习题 ➔ 尝试构建项目。

而在 AI 时代，我认为它应当彻底倒过来：

动手构建小东西 ➔ 借力 AI 跨越障碍 ➔ 深度审查产出的代码 ➔ 提炼并吃透核心概念 ➔ 迁移复用模式 ➔ 在需要时自然扩张领土。

这几乎是完全相反的。 它不是纯粹的自上而下，也不是纯粹的自下而下，它更像是一个螺旋上升的晶体结构。 你从一个对你有意义的具体任务开始。 你学习了足够的核心语言知识，以便能看懂刚才到底发生了什么。 你带着这些知识，走向下一个任务。 你归纳总结出规律。 你丰富了自己的工具箱。 你进行代码审查。 你进行提炼压缩。 你再次出发。

这完全契合一个成年人学习现实技能的真实客观规律。

The deepest shift

In the pre-AI era, learning a programming language often meant:

Learn syntax → do exercises → build projects.

In the AI era, I think it should become:

Build small things → use AI to cross obstacles → inspect the code → extract concepts → reuse patterns → expand when needed.

That is almost the reverse.

Not pure top-down. Not pure bottom-up. More like a spiral.

You start with a meaningful task.
You learn enough core language to understand what happened.
You return to another task.
You generalize.
You add tools.
You review.
You compress.
You repeat.

This fits how adults actually learn.

最终回答

你提出的“基础工具箱”方向完全正确，但我建议将其升维成一个三层的体系：

  • 核心箱（Core case）：语言底层的通用基石。

  • 目的箱（Purpose case）：为了满足你当下的真实商业或研究目的而需要的特定工具。

  • 扩张箱（Expansion case）：让你能够独立上手新库、阅读文档、调试报错，在未来成长中绝不迷失方向的方法论。

AI 时代的反馈回路不应该是：

记笔记 + 做死题 + 死记硬背。

它应当是：

实际需求 ➔ 微型构建 ➔ AI 协助 ➔ 代码研读 ➔ 故障调试 ➔ 文档查阅 ➔ 迁移复用 ➔ 提炼笔记 ➔ 下一次构建。

同时，学习一门编程语言的核心指标，再也不应该是：

“我学完了 Python 的所有语法。”

它应当变成：

“我能把 Python 当成一个完全受我控制的工具来使用；并且当我遇到全新的东西时，我明确知道该如何去快速吸收它。”

这，才是我认为 AI 带给我们的、真正的解放。 并不是说我们不再需要学习了。 而是我们终于可以停止假装编程的核心就是去死记硬背那些琐碎的操作方法，从而得以重新回归到那个真正迷人的本质：

结构、拆解、抽象、测试、调试、审查，以及亲手去构建出对这个现实世界真正重要的东西。

Final answer

Your proposed “basic tool case” is the right direction, but I would make it three-layered:

Core case: the universal foundation of the language.
Goal case: the tools needed for your actual purpose.
Expansion case: the ability to learn new libraries, read docs, debug, and grow without feeling lost.

The AI-era feedback loop should not be:

notes + exercises + memorization.

It should be:

real need → small build → AI assistance → code reading → debugging → documentation lookup → reuse → compressed notes → next build.

And the goal of learning a programming language should no longer be:

“I have covered Python.”

It should be:

“I can use Python as a controllable tool, and when I meet something new, I know how to absorb it.”

That, to me, is the real liberation AI gives us.

Not that we no longer need to learn.

But that we can finally stop pretending the main point of programming is memorizing tiny methods, and return to the actual core:

structure, decomposition, abstraction, testing, debugging, review, and building things that matter.
